# 輕量模型推薦（8GB 以下 VRAM）

[[AI 101 - 主頁|← 回主頁]]

> [!tip] 適用情境
> 硬體記憶體較小（8GB VRAM 以下），或想要推理速度快、低延遲的場景。
> 以下模型皆可透過 Ollama 直接安裝，一行指令搞定。

---

## 快速選型指南

| 模型 | 大小 | VRAM | 速度 | 最強項目 | 一行安裝 |
|---|---|---|---|---|---|
| **Phi-4 Mini** | 3.8B | ~3 GB | 28 tok/s | 推理、數學、程式碼 | `ollama pull phi4-mini` |
| **Gemma 3:4b** | 4B | ~4.2 GB | 快 | 多語言、圖片理解 | `ollama pull gemma3:4b` |
| **Qwen3:4b** | 4B | ~4 GB | 快 | 推理、中文、程式碼 | `ollama pull qwen3:4b` |
| **Llama 3.2:3b** | 3B | ~2.5 GB | 極快 | 輕量任務、低延遲 | `ollama pull llama3.2:3b` |
| **Qwen3-Coder** | 多版本 | 依版本 | 快 | 程式碼專用 | `ollama pull qwen3-coder-next` |

---

## 各模型詳細說明

### Phi-4 Mini（Microsoft）

> **2026 年 8GB 以下綜合評比第一名**

```bash
ollama pull phi4-mini
ollama run phi4-mini
```

**規格：**
- 參數：3.8B
- VRAM：~3 GB
- 速度：28 tok/s（8GB 系統）

**Benchmark（vs Llama 3.2 3B）：**

| 項目 | Phi-4 Mini | Llama 3.2 3B |
|---|---|---|
| MMLU | **73%** | 65% |
| MATH | **62%** | 48% |

**最適合：** 推理、數學、程式碼、需要較高品質但硬體受限的任務。

---

### Gemma 3:4b（Google）

> **記憶體最省、支援圖片、128K context**

```bash
ollama pull gemma3:4b
ollama run gemma3:4b
```

**規格：**
- 參數：4B
- VRAM：**~4.2 GB**（同類最省）
- Context：**128K tokens**
- 支援語言：140+
- 多模態：支援圖片輸入

**最適合：** 記憶體非常有限、需要長 context、多語言環境、圖片分析。

> [!tip]
> 如果你的 VRAM 剛好 8GB，想跑 7B 以上的模型又塞不下，
> Gemma 3:4b 的 128K context 可以彌補參數量的不足。

---

### Qwen3:4b（阿里巴巴）

> **推理能力遠超同參數量，中文最強**

```bash
ollama pull qwen3:4b
ollama run qwen3:4b
```

**規格：**
- 參數：4B
- VRAM：~4 GB
- 支援：思考模式（Chain of Thought）

**亮點：** Qwen3-4B 的推理能力可以媲美 Qwen2.5-72B-Instruct——同樣 4B 的參數做到了 72B 的水準。

**最適合：** 中文任務、邏輯推理、需要「思考」的複雜問題。

> [!info] 思考模式開啟方式
> 在 prompt 中加入 `/think` 指令，或在對話中要求「請一步一步思考」。

---

### Llama 3.2:3b（Meta）

> **最快、最輕，零延遲體驗**

```bash
ollama pull llama3.2:3b
ollama run llama3.2:3b
```

**規格：**
- 參數：3B
- VRAM：~2.5 GB
- 速度：極快（幾乎感覺不到延遲）

**最適合：** 低延遲需求、簡單任務（格式轉換、文法檢查、短文摘要、CSV 轉 Markdown）。

> [!warning] 能力上限較低
> 適合「低風險」的簡單任務。複雜推理、程式碼生成建議用 Phi-4 Mini 或 Qwen3:4b。

---

### Qwen3-Coder（阿里巴巴）

> **程式碼專用，Agentic 工作流程優化**

```bash
ollama pull qwen3-coder-next
ollama run qwen3-coder-next
```

**特色：**
- 專為 agentic coding workflow 設計
- 優化工具呼叫（tool calling）
- 針對本地開發場景調校

**最適合：** 接 OpenClaw / Hermes Agent 做程式碼任務、自動化開發流程。

---

## 量化格式說明

Ollama 模型名稱後面可以加量化版本：

```bash
ollama pull qwen3:4b          # 預設（通常是 Q4_K_M）
ollama pull qwen3:4b-q4_K_M   # 明確指定 Q4_K_M
ollama pull qwen3:4b-q8_0     # 更高精度，需要更多 VRAM
```

| 量化 | 品質 | VRAM 需求 | 建議 |
|---|---|---|---|
| Q4_K_M | ~95% 原始能力 | 最低 | **一般推薦** |
| Q6_K | ~98% | 中 | VRAM 夠用時選這個 |
| Q8_0 | ~99% | 較高 | 接近完整品質 |
| F16 | 100% | 最高（2x）| 通常沒必要 |

> [!tip] Q4_K_M 是甜蜜點
> 壓縮 9B 模型到約 5GB，保留 95% 能力，完整 GPU offload，推薦作為預設選擇。

---

## 依 VRAM 大小選模型

| 可用 VRAM | 推薦模型 | 說明 |
|---|---|---|
| < 4 GB | `llama3.2:3b` | 能跑就好，速度快 |
| 4–6 GB | `phi4-mini` / `gemma3:4b` / `qwen3:4b` | 甜蜜點，CP 值最高 |
| 6–8 GB | `qwen3:8b` / `llama3.1:8b` | 40+ tok/s，大多數任務夠用 |
| 8–16 GB | `gemma4:e4b` + 以上 | 可以考慮 gemma4 系列 |

---

## 接上 OpenClaw / Hermes Agent

確認 context 夠用後（參考 [[AI 101 - Gemma 4 本地模型#Step 2：確認 Context 大小是否達到 64K]]），設定方式與 Gemma 4 相同：

**OpenClaw**（`~/.openclaw/openclaw.json`）：
```json
{
  "model": {
    "provider": "ollama",
    "baseUrl": "http://<IP>:11434",
    "model": "qwen3:4b",
    "contextWindow": 65536
  }
}
```

**Hermes Agent**（`hermes setup`）：
```
provider: ollama
base URL: http://<IP>:11434/v1
model:    qwen3:4b
```
